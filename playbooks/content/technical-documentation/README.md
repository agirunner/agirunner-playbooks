# Technical Documentation

## Use When

Use this playbook when technical facts or workflows need to become a
reader-focused document rather than a loose internal note.

## Not Ideal When

Do not use this playbook for pure API reference material, changelog or
release-note writing, or broad product marketing copy. Import
`api-documentation` when the main job is to describe request and
response contracts rather than help a reader complete a task.

## Inputs

- documentation goal
- target reader
- source material context

## Output

The workflow should produce:
- target reader and document goal
- document type and source-truth basis
- drafted documentation
- important limits
- validation questions if the source truth is incomplete
- editorial review findings
- follow-up documentation gaps

## Workspace Notes

This playbook works with a writable documentation workspace or in
advisory mode. A writable workspace improves the handoff but is not
required for honest completion.

## Import Notes

This playbook is strongest when the imported specialists can access the
same source truth the human author would use. When that source truth is
partial, stale, or conflicting, the authored workflow should still close
with a bounded draft and explicit validation questions rather than
invented certainty.
